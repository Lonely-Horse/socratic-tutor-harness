### Assistant:
**[观察]**
你开始关注文件描述符（FD）泄露，这是走向生产级 Go 开发的关键一步。FD 泄露通常隐蔽地发生在大流量或长时间运行的服务中，最终会导致经典的 `too many open files` 错误，造成服务宕机。

**[概念]**
在 Go 中，文件、网络连接（TCP socket）、数据库连接等，在底层都是操作系统分配的文件描述符。Go 提供了 `io.Closer` 接口，要求我们显式地释放这些资源。垃圾回收器（GC）**不会**自动帮我们即时关闭它们，必须由开发者主动管理。

---

**[引导]**
为了写出不泄露 FD 的代码，请思考以下三个在 Go 中极易犯错的典型场景：

### 场景一：文件操作与错误处理
通常我们使用 `os.Open` 打开文件。
1. 你会把 `defer file.Close()` 放在 `err != nil` 的检查之前还是之后？为什么？
2. 如果 `os.Open` 失败了（返回了 error），此时的 `file` 指针是什么？对其调用 `Close()` 会发生什么？

### 场景二：HTTP 请求的响应体
使用标准库 `net/http` 发送请求时：
```go
resp, err := http.Get("https://api.example.com")
// 应该在这里 defer resp.Body.Close() 吗？
```
1. 如果 `http.Get` 返回了 `err`（比如网络超时），此时 `resp` 是否为 `nil`？如果直接 `defer resp.Body.Close()` 会有什么风险？
2. 如果请求成功了（`err == nil`），但你只读取了部分数据，或者根本没有读取数据，只是调用了 `Close()`，这个网络连接能被复用吗？

### 场景三：循环体内的资源释放
如果在 `for` 循环中频繁打开文件或建立连接：
```go
for _, path := range filePaths {
    f, err := os.Open(path)
    if err != nil {
        continue
    }
    defer f.Close() // 这样写会有什么问题？
    // 处理文件...
}
```
1. `defer` 的执行时机是“当前函数退出时”还是“当前代码块（如 for 循环）结束时”？
2. 如果 `filePaths` 有上万个文件，这段代码在运行期间会占用多少个文件描述符？你会如何重构它以避免堆积？
